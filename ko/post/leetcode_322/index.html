Coin Change (동전 계산하기) 문제로 동적 계획법 이해하기

문제 소개

리트코드 원본 문제 : (리트코드 사이트)

You are given an integer array coins representing coins of different denominations and an integer amount representing a total amount of money.
Return the fewest number of coins that you need to make up that amount. If that amount of money cannot be made up by any combination of the coins, return -1. You may assume that you have an infinite number of each kind of coin.

이를 한국어로 해석해보자면,

입력으로 “단위가 다른 동전 종류(어레이)“와 “동전들을 가지고 만들 총 금액” 두가지를 받습니다. 이때, 총 금액 값을 만드는 가장 적은 동전의 갯수를 반환하십시오. 만약 가지고 있는 동전들로 총 금액을 만들 수 없는 경우에는 -1을 반환하며, 각 동전의 개수는 항상 총액을 만들 수 있을 만큼 충분히 많다고 가정합니다.

이 문제는 책 초반에 다양한 PS 문제들의 기본적인 접근법이 되는 프레임들을 안내해주는 1.2장에서 소개 되었습니다. 저는 아직 알고리즘 문제를 풀어보는 경험이 아에 없다시피 하기 때문에 쉽게 설명하고 넘어가는 부분에도 오랜시간을 보내며 어떻게 작동하는지 그려보며 따라가 보았습니다!

책에서 제시하는 키워드는 DP, 동적 계획법으로의 접근입니다.

접근법

동적 계획법 [Dynamic Programming]

동적 계획법의 개념 자체는 어렵고 복잡한 문제를 간단한 여러개의 작은 문제들로 나누어 풀이해내는 방법입니다. 분명 개념은 아주 쉽죠..? 하지만 알고리즘 문제를 풀면서 동적 계획법을 써보세요~ 하면 머릿속이 복잡해기 시작할겁니다.

너무 간단한 개념 설명을 조금 보충할 필요가 있을 것 같아서 위키 설명을 참조해 보겠습니다!

(동적 계획법이란) 주어진 문제를 풀기 위해서, 문제를 여러 개의 하위 문제(subproblem)로 나누어 푼 다음, 그것을 결합하여 최종적인 목적에 도달하는 것이다. 각 하위 문제의 해결을 계산한 뒤, 그 해결책을 저장하여 후에 같은 하위 문제가 나왔을 경우 그것을 간단하게 해결할 수 있다. 이러한 방법으로 동적 계획법은 계산 횟수를 줄일 수 있다. 특히 이 방법은 하위 문제의 수가 기하급수적으로 증가할 때 유용하다.

자, 그럼 개념을 통해서 동전 계산하기 문제를 봅시다. 지금 주어진 큰 문제를 작은 여러개의 하위 문제로 나눌 필요가 있습니다. 정말 단순하게 떠오르는 접근법으론 총 금액에서 가지고 있는 종류의 동전을 하나씩 빼보면서 정확히 0을 맞추어 낼 수 있는지를 확인하면 될 것 같네요. 그렇다면 여기서 ‘작은 문제’는 ‘총 금액에서 종류별 금액을 빼보기’가 되겠습니다. 그럼 총 금액을 인자로 받아 가지고 있는 동전의 종류마다 하나씩 빼는 함수를 짜봅시다!

여기 하위 문제를 푸는 함수 dp(n)을 만들었습니다.
남은 금액 n을 인자로 받고, 가지고 있는 종류의 동전(coin)마다 하나씩 뺀 뒤에 재귀적으로 다시 dp(n - coin)을 넣습니다.

위 함수는 동전을 뺐을 때 금액이 0이 되는 경우나 음수가 되는 예외적인 경우를 아직 포함하지 않았네요.

result = min(result, 1 + dp(n - coin))

위 코드가 오늘 동적 계획법 적용의 핵심입니다. 재귀적으로 dp(n - coin)을 부름으로써 coin을 빼는 모든 경우를 계산합니다. 그 뒤엔 이를 각각 지금의 result값과 비교해서 더 적은 값(최소 값)을 돌려주도록 설계했습니다. (저는 이 짧은 부분을 이해하는데 한참이 걸린 것 같네요.)

이제 해당 문제를 실제로 풀이하기 위해 코드를 수정해봅시다. Solution 클래스와 coinChange 함수는 LeetCode에서 제공하는 기본 템플릿이며, 문제를 풀기 위해서는 이 내부에 입력값과 출력값을 고려하여 코드를 작성하면 됩니다.

class Solution(object):
    def coinChange(self, coins, amount):
        """
        :type coins: List[int]
        :type amount: int
        :rtype: int
        """
        def dp(n): 
            # 가장 기본적인 케이스(base case)
            if n == 0: return 0
            if n < 0: return -1
            # 최솟값을 구할 때 비교하기 위해 가장 큰 값인 양의 무한대로 초기화 ()
            result = float('INF')
            for coin in coins:
                subproblem = dp(n - coin)
                # 하위 문제의 답이 없을 시 -1을 리턴하므로 건너뛰기
                if subproblem == -1: continue
                result = min(result, 1 + subproblem)
            return result if result != float('INF') else -1

        return dp(amount)

문제 해결을 위해서 몇가지를 수정하긴 했지만, 핵심 구조는 동일합니다!
개인적으로 바로 이해가 안됬던 부분은 result = float('INF') 이 부분이었는데, 차근히 생각해보니 이해가 됩니다. 함수의 핵심 구조는 result = min(result, 1 + dp(n - coin)) 이고 여기서 min()을 이용하여 최솟값을 누적하여 구하고 있습니다. 이렇게 되면 처음 초기화 한 result의 값이 min()에 들어갔을 때 나올 수 있는 최솟값 보다 항상 커야만 하며, 그러므로 정의할 수 있는 가장 큰 값인 양의 무한대로 초기화를 해야 어떠한 예외없이 result의 초기값이 항상 크다는걸 보장할 수 있게 됩니다.

동적 계획법 개선하기

위의 방식은 단순 재귀함수를 이용한 접근으로 동적계획법을 잘 이용하긴 했지만 같은 계산을 중복으로 하게 되는 문제가 있습니다.
예를 들어 coins = [2,3,5], amount = 17 이 입력되었을 때, 위 함수는 17 - 5 - 5 - 2 - 5 = 0 와17 - 5 - 2 - 5 - 5 = 0를 계산하는 경우애 첫 dp(17 - 5) 값을 매번 일일히 계산하고 나서야 값을 반환합니다. 하지만 만약 이 dp(12) 값을 어딘가 저장해두고 다시 찾을 때 반환한다면 중복 없이 훨씬 빠르게 문제를 끝낼 수 있게 될겁니다! 이처럼 저장해두는 장소를 보통 메모라고 선언 한다. 아래는 메모를 이용한 동적 계획법 풀이를 적어보았습니다.

class Solution(object):
    def coinChange(self, coins, amount):
        """
        :type coins: List[int]
        :type amount: int
        :rtype: int
        """
        memo = {}
        def dp(n):  #need to find base case for result
            if n in memo: return memo[n] # 메모에 값이 있으면 바로 반환
            if n == 0: return 0
            if n < 0: return -1
            result = float('INF')
            for coin in coins:
                sub = dp(n - coin)
                if sub == -1: continue
                result = min(result, 1 + sub)
            memo[n] = result if result != float('INF') else -1
            return memo[n]

        return dp(amount)

memo라는 변수를 딕셔너리 형태로 선언하고 계산하기 전에 저장된 메모 값을 먼저 확인하여 중복된 계산을 바로 피할 수 있게 되었습니다!
